# ロール・リソース管理 仕様整理書

最終更新: 2026-06-16
対象: AI-PMO（FastAPI バックエンド + Vue3/Vuetify フロントエンド）

---

## 0. 本書の目的

リソース管理（ユーザー管理）まわりの仕様が混乱した状態で実装されているため、本書で**正しい仕様に整理**する。本書を基に、別途 Antigravity への実装指示文書を作成する。

主な是正点は次の2つ:

1. **「操作ユーザー切替」機能の廃止** — ロールはログイン時に確定するため、セッション中の切替は不要かつ危険（誰でも管理者になれてしまう）。**メール+パスワード認証によるログイン**へ置き換える。
2. **システム管理者（`system_role = 管理者`）はリソースとしてアサイン不可** — 管理者はアサイン候補（NLP推薦・各種アサインUI）から除外し、**バックエンドでもアサインを拒否**する。

## 1. 現状の問題点（確認結果）

| # | 現状 | 問題 |
|---|---|---|
| 1 | 認証基盤が無い。サイドバーの「操作ユーザー切替」ドロップダウン（`App.vue`）がログインの代用。`store.currentRole` 既定値が `管理者` | 誰でも自由に管理者へ切替可能。権限管理が意味を成さない |
| 2 | 権限判定をフロントが送る `X-User-Role` / `X-User-Id` ヘッダで実施（`store/project.js` の axios interceptor、`resources.py` の `x_user_role`） | クライアントが自分の権限を自己申告 = 簡単に詐称可能 |
| 3 | `system_role`（管理者/マネージャ/メンバー＝システム権限）と `role`（PG/SE/PM/SV＝職務役割）の2概念が混在し、UI上の区別が曖昧 | 用語・表示の整理が必要 |
| 4 | NLPリソース推薦・アサイン候補に `system_role = 管理者` のユーザーも含まれる | 管理者が作業リソースとして選定されうる |
| 5 | タスク担当 `assignee_name` は自由入力（`GanttView.vue`）。リソースマスタと未連携 | アサインの一貫性・除外制御が効かない（将来課題として明記） |

## 2. 用語・概念の整理（重要）

2種類の「ロール」を明確に分離する。UI文言・項目名もこれに統一する。

| 概念 | フィールド | 値の例 | 用途 |
|---|---|---|---|
| **システム権限** | `system_role` | `管理者` / `マネージャ` / `メンバー` | 画面操作・データ操作の権限制御、ログイン後のロール |
| **職務ロール** | `role` | `PM` / `SV` / `SE` / `PG` 等 | スキル・職種。アサインやNLP推薦の判定材料 |

- UI表記:システム権限は「システム権限」、職務ロールは「役割（職種）」と表示する。
- 「リソース」=「ユーザー」=`resources` テーブルの1レコード。本書では「リソース」に統一。

## 3. 認証・ログイン仕様（メール+パスワード）

### 3.1 方針
- ロールは**ログイン時に確定**し、セッション中は**変更不可**。
- 「操作ユーザー切替」UIは**完全廃止**。
- バックエンドは**認証トークンから system_role を導出**する。クライアント送信の `X-User-Role` ヘッダは権限判定に**使用しない**（廃止）。

### 3.2 ログインフロー
1. 未認証ユーザーは `/login` 画面へリダイレクト。
2. メールアドレス + パスワードを入力 → `POST /api/v1/auth/login`。
3. 成功時、サーバはアクセストークン（JWT）+ ログインユーザー情報（id / name / email / system_role / role）を返す。
4. フロントはトークンを保持（`localStorage` 等）し、以降のリクエストに `Authorization: Bearer <token>` を付与。
5. `system_role` を `store.currentRole` に格納（**読み取り専用**、変更アクションは設けない）。
6. ログアウトボタンでトークン破棄 → `/login` へ。

### 3.3 トークン/セッション
- 方式: JWT（署名秘密鍵は `settings.SECRET_KEY` を新設、環境変数化）。
- ペイロード: `sub`(resource_id), `system_role`, `exp`。
- 有効期限: 既定 8〜12時間程度（設定値化）。
- `is_active = false` のリソースはログイン拒否。

### 3.4 パスワード管理
- `resources` に `password_hash` を追加（bcrypt 等、平文保存禁止）。
- 管理者によるリソース新規作成時に初期パスワードを設定 or 招待メール…はスコープ外。**当面は管理者がパスワードを設定/リセットできる**形とする（`PUT /resources/{id}` で `password`（平文受取→ハッシュ化保存）を受け付ける。レスポンスには `password_hash` を**含めない**）。
- シードユーザーには初期パスワードを設定（下記 9 章）。

### 3.5 既存ヘッダ方式の置換
- `resources.py` の `x_user_role: Header(...)` を廃止し、**認証依存（`get_current_user`）**から `system_role` を取得する方式へ統一。
- フロント axios interceptor の `X-User-Role` / `X-User-Id` 付与を廃止し、`Authorization` ヘッダ付与へ変更。
- 認証必須エンドポイントは共通依存 `Depends(get_current_user)` で保護。`/auth/login` のみ未認証で許可。

## 4. ロール別権限マトリクス

| 操作 | 管理者 | マネージャ | メンバー |
|---|:--:|:--:|:--:|
| ログイン | ○ | ○ | ○ |
| リソース閲覧（一覧/詳細） | ○ | ○ | ○ |
| リソース新規作成 | ○ | × | × |
| リソース編集（一般項目） | ○ | ○ | × |
| リソースの `system_role` 変更 | ○ | × | × |
| リソースの `is_active` 切替 | ○ | × | × |
| リソース削除 | ○ | × | × |
| 時間単価 `hourly_cost_jpy` の閲覧 | ○ | ○ | ×（マスク=0表示） |
| パスワード設定/リセット | ○ | ×（自分のみ任意） | ×（自分のみ任意） |
| カレンダー（祝日）管理 | ○ | ○ | × |
| **リソースとしてアサインされる対象になるか** | **×（不可）** | ○ | ○ |

> 注: マスク等の既存挙動（`mask_resource_cost`）は維持。判定元をヘッダからトークン由来 role に変更するのみ。

## 5. 管理者のリソースアサイン禁止仕様

`system_role = 管理者` のリソースは「作業リソース」ではなく管理者であるため、**アサイン対象から除外**する。適用範囲は **UI候補除外 + バックエンド拒否の両方**。

### 5.1 バックエンド（強制）
- **アサイン候補返却系**は `system_role = 管理者` を除外して返す:
  - `GET /api/v1/resources/search`（NLP推薦）→ 候補生成前に管理者を除外（LLM入力リストからも除外）。
  - （新設する場合の）アサイン候補取得API も同様に除外。
- **アサイン確定系**は管理者を指定された場合 `400 Bad Request` を返す:
  - `ResourceAllocation` 作成/更新で `resource_id` が管理者を指す場合は拒否（エラーメッセージ例:「システム管理者はリソースとしてアサインできません。」）。
  - タスク担当をリソース連携する場合（将来 `assignee` をリソース参照にする場合）も同様に拒否。
- 除外判定は共通ヘルパ（例: `is_assignable(resource) -> bool := resource.system_role != "管理者" and resource.is_active`）に集約。

### 5.2 フロントエンド（候補非表示）
- NLP推薦結果、アサイン用セレクト/サジェスト等、**アサイン候補を提示する全UIで管理者を非表示**。
- ただし**リソース一覧（マスタ）画面（AdminView）では管理者も表示**する（彼らはユーザーとして管理対象であるため）。一覧上で「アサイン不可」が分かるよう視覚表現（バッジ/淡色表示/「アサイン対象外」注記）を付与。

### 5.3 AIワーカーの扱い
- AIワーカーは `system_role = メンバー` のリソースであり、**アサイン対象として存続**（除外対象は管理者のみ）。

## 6. UI/UX 整理

### 6.1 サイドバー（`App.vue`）
- **削除**: 「操作ユーザー切替」セレクト（`selectedUser` / `userOptions` / `setCurrentUser` 連動）一式。
- **追加**: サイドバー下部に**ログイン中ユーザー表示**（氏名 + システム権限バッジ）+ **ログアウトボタン**。権限バッジは読み取り専用。
- 「システム管理」ナビ項目は `system_role = 管理者`（およびマネージャは一部のみ）に応じて表示制御（メンバーには非表示 or 閲覧のみ）。

### 6.2 ログイン画面（新規 `Login.vue` / ルート `/login`）
- メール / パスワード入力、ログインボタン、エラー表示（認証失敗・無効アカウント）。
- ログイン済みで `/login` に来た場合はダッシュボードへリダイレクト。
- 認証ガード（router beforeEach）で未認証アクセスを `/login` へ。

### 6.3 システム管理画面（`AdminView.vue`）
- リソース一覧:管理者行も表示しつつ「アサイン対象外」を明示。
- 編集ダイアログ:`system_role` / `is_active` は管理者のみ編集可（既存の `:disabled` 条件は維持、判定元を `store.currentRole` の**ログイン由来値**に）。
- 「役割（職種）」と「システム権限」のラベルを明確化（用語整理に準拠）。
- パスワード設定欄を追加（管理者は任意のユーザー、本人は自分のパスワードを変更可）。
- **NLP推薦パネル**:候補に管理者を出さない（5.2）。

### 6.4 タスク作成（`GanttView.vue`）
- 現状の `assignee_name` 自由入力は当面維持可。ただし**将来的にリソースマスタ選択へ移行**することを推奨（その際は管理者を候補から除外）。本対応では「リソース選択にした場合は管理者除外」を仕様として明記しておく。

## 7. データモデル変更（`resources` テーブル）

| 変更 | 内容 |
|---|---|
| 追加 | `password_hash: str`（NOT NULL、bcrypt ハッシュ。シードで初期値設定） |
| 変更 | `email` を **unique + index** に（ログインIDとして使用） |
| 追加（任意） | `last_login_at: datetime nullable`（運用補助） |

- `system_role` の取りうる値を `管理者 / マネージャ / メンバー` に限定（バリデーション）。
- スキーマ（`schemas/resource.py`）: `ResourceResponse` に `password_hash` を**含めない**。`ResourceCreate`/`ResourceUpdate` に `password`（平文・任意）を受け付け、サーバでハッシュ化。

## 8. API 変更まとめ

### 8.1 新設
- `POST /api/v1/auth/login` — body: `{email, password}` → `{access_token, token_type, user:{id,name,email,system_role,role}}`
- `GET /api/v1/auth/me` — 認証必須。現在のユーザー情報を返す。
- （任意）`POST /api/v1/auth/logout` — クライアント側トークン破棄で代替可なら省略可。

### 8.2 変更
- 全保護エンドポイント: `x_user_role: Header` を廃止し `current = Depends(get_current_user)` に統一。権限判定は `current.system_role`。
- `GET /resources/search`: 管理者を候補から除外（LLM入力・結果の双方）。
- `ResourceAllocation` 作成/更新系（`pmo.py` のアロケーション操作含む）: 管理者 `resource_id` を拒否（400）。
- フロント `store/project.js`: interceptor を `Authorization: Bearer` 方式へ。`currentRole` はログイン由来・読み取り専用。`setCurrentUser`（切替）アクションを削除し、`login` / `logout` / `fetchMe` を新設。

### 8.3 依存追加（`requirements.txt`）
- `passlib[bcrypt]`（パスワードハッシュ）
- `pyjwt`（または `python-jose[cryptography]`）（JWT発行/検証）
- 設定 `SECRET_KEY` / `ACCESS_TOKEN_EXPIRE_HOURS` を `config.py` に追加（環境変数化）。

## 9. シード/移行

- 既存シードリソース（`main.py`）に `password_hash` を付与:
  - `nhigashira`（管理者）／ Manager（マネージャ）／ Frontend（メンバー）／ AI Worker（メンバー）。
  - 初期パスワードは開発用の既知値（例: `password` など）をハッシュ化して投入。値は仕様確定時に決定。
- `email` を一意にするため、シードの email 重複が無いことを確認。
- 既存DBに `resources` がある場合のマイグレーション（`password_hash` 追加 + 既存行へ初期ハッシュ補填）方針を Antigravity 指示に含める（現状は起動時 `create_all` のため、列追加は手動対応が必要な点に留意）。

## 10. 受け入れ条件（Acceptance Criteria）

1. 「操作ユーザー切替」UI が存在しない。サイドバーにログイン中ユーザー + ログアウトのみ表示。
2. 未認証で各画面にアクセスすると `/login` にリダイレクトされる。
3. 正しいメール+パスワードでログインでき、`system_role` がセッション中固定される（変更手段が無い）。
4. 誤ったパスワード / `is_active=false` ユーザーはログインできない。
5. バックエンドの権限判定がトークン由来 role で行われ、クライアントの `X-User-Role` ヘッダでは権限が変わらない（詐称不可）。
6. NLP推薦・アサイン候補に `system_role=管理者` が出ない。
7. `ResourceAllocation` 等で管理者を `resource_id` 指定すると 400 で拒否される。
8. リソース一覧（マスタ）では管理者も表示され、「アサイン対象外」が判別できる。
9. メンバーには `hourly_cost_jpy` が 0 マスク表示される（既存挙動維持、判定元のみ変更）。
10. AIワーカー（メンバー）は従来どおりアサイン候補に出る。

## 11. 本対応のスコープ外（将来課題として記録）

- タスク `assignee_name` 自由入力のリソースマスタ参照化（FK化）。
- `ResourceAllocation.resource_name` と `resource_id` の整合（名称の二重管理解消）。
- 招待メール/パスワードリセットメール等の本格的アカウント運用。
- リフレッシュトークン・監査ログ等の高度なセキュリティ機能。
